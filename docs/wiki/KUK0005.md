## 💡 Title
Kuk0005TagWithCallSiteOnExecutionAnalyzer

## 🆔 Rule ID
KUK0005

## 📝 Description
Require [.TagWithCallSite()](https://learn.microsoft.com/en-us/dotnet/api/microsoft.entityframeworkcore.entityframeworkqueryableextensions.tagwithcallsite?view=efcore-10.0) on EF Core query execution

Entity Framework Core queries should be annotated with `.TagWithCallSite()` before materialization or execution (e.g., `ToList`, `First`, `Count`, `ExecuteUpdate`).

This helps improve observability, debugging, and tracing of generated SQL by including call site information.

## ❔ Why this is a problem
Developers may forget to call this method, and it cannot be configured globally ([why?](https://stackoverflow.com/a/70658360/32494813)).

## ⚙️ Configuration

You can configure the severity of the rule using .editorconfig.
```.editorconfig
[*.cs]
dotnet_diagnostic.KUK0005.severity = warning # .TagWithCallSite() on EF Core query execution | https://github.com/kurnakovv/kuker/wiki/KUK0005
```

## 💻 Examples
```cs
// ❌ KUK0005: Use .TagWithCallSite() for 'ToListAsync' method
List<User> adultsUsersViolation = await _appDbContext.Users
    .Where(x => x.Age >= 18)
    .ToListAsync();

// ✅ OK
List<User> adultsUsersOK = await _appDbContext.Users
    .TagWithCallSite() // <- Because we use .TagWithCallSite() here
    .Where(x => x.Age >= 18)
    .ToListAsync();
```

## 🗒️ Notes
* Placement of `.TagWithCallSite()` | It does not matter where `.TagWithCallSite()` appears in the query chain (beginning, middle, or right before execution). The only requirement is that it must be part of the same query chain that ends with the executing method.

* What counts as an executing method | All EF Core executable methods from the official Microsoft [documentation](https://learn.microsoft.com/en-us/dotnet/api/microsoft.entityframeworkcore.entityframeworkqueryableextensions). Executable methods are those that are translated into a database query (such as `ToList`, `FirstOrDefault`, etc)

* `.TagWithCallSite()` inside another method (incorrect) | If `.TagWithCallSite()` is applied inside a method that returns `IQueryable`, it captures the creation site, not the execution site.

    Example
    ```cs
    using Microsoft.AspNetCore.Mvc;
    using Microsoft.EntityFrameworkCore;

    namespace JustApplicationWebApi.Controllers;

    [Route("api/[controller]")]
    [ApiController]
    public class TagWithCallSiteController : ControllerBase
    {
        private readonly AppDbContext _appDbContext;

        public TagWithCallSiteController(
            AppDbContext appDbContext)
        {
            _appDbContext = appDbContext;
        }

        [HttpGet("Violation")]
        public async Task<IActionResult> Violation()
        {
            var users = await GetUserQuery().ToListAsync(); // 21 line.

            return Ok(users);
        }

        [HttpGet("Good")]
        public async Task<IActionResult> Good()
        {
            var users = await _appDbContext.Users.TagWithCallSite().ToListAsync(); // 29 line.

            return Ok(users);
        }

        private IQueryable<User> GetUserQuery()
        {
            return _appDbContext.Users.TagWithCallSite(); // 36 line.
        }
    }
    ```

    Call `TagWithCallSiteController.Violation`:

    ```
    -- File: D:\C#\Asp Net Core\JustApplicationWebApi\JustApplicationWebApi\Controllers\TagWithCallSiteController.cs:36

    SELECT u.id_user, u.u_deleted_date, u.u_name, u.u_phone
    FROM "user" AS u
    ```

    Call `TagWithCallSiteController.Good`:

    ```
    -- File: D:\C#\Asp Net Core\JustApplicationWebApi\JustApplicationWebApi\Controllers\TagWithCallSiteController.cs:29

    SELECT u.id_user, u.u_deleted_date, u.u_name, u.u_phone
    FROM "user" AS u
    ```

    As shown above, the first call points to line 36, which is problematic. For example, if the `GetUserQuery` method is located in another file and reused in multiple places, it becomes very difficult to determine where the query was actually executed.

    Therefore, `.TagWithCallSite()` should be applied at the point of execution.

* `.TagWithCallSite()` on a query variable (incorrect) | If `.TagWithCallSite()` is used when defining a variable, and execution happens later when that variable is used (for example, `var query = _db.Users.TagWithCallSite();` + `await query.ToListAsync()`), then this rule is violated.

    Example
    ```cs
    int age = 10;
    var query = _appDbContext.Users.TagWithCallSite(); // 30 line.

    var users1 = await query.Where(x => x.Age >= age).ToListAsync();

    // some work with age
    age = 15;

    var users2 = await query.Where(x => x.Age >= age).ToListAsync();

    // some work with age
    age = 30;

    var users3 = await query.Where(x => x.Age >= age).ToListAsync();
    ```

    ```
    Microsoft.EntityFrameworkCore.Database.Command: Information: Executed DbCommand (46ms) [Parameters=[@__age_0='?' (DbType = Int32)], CommandType='Text', CommandTimeout='30']
    -- File: D:\C#\Asp Net Core\JustApplicationWebApi\JustApplicationWebApi\Controllers\TagWithCallSiteController.cs:30

    SELECT u.id_user, u.u_age, u.u_deleted_date, u.u_name, u.u_phone
    FROM "user" AS u
    WHERE u.u_age >= @__age_0


    Microsoft.EntityFrameworkCore.Database.Command: Information: Executed DbCommand (6ms) [Parameters=[@__age_0='?' (DbType = Int32)], CommandType='Text', CommandTimeout='30']
    -- File: D:\C#\Asp Net Core\JustApplicationWebApi\JustApplicationWebApi\Controllers\TagWithCallSiteController.cs:30

    SELECT u.id_user, u.u_age, u.u_deleted_date, u.u_name, u.u_phone
    FROM "user" AS u
    WHERE u.u_age >= @__age_0


    Microsoft.EntityFrameworkCore.Database.Command: Information: Executed DbCommand (3ms) [Parameters=[@__age_0='?' (DbType = Int32)], CommandType='Text', CommandTimeout='30']


    -- File: D:\C#\Asp Net Core\JustApplicationWebApi\JustApplicationWebApi\Controllers\TagWithCallSiteController.cs:30

    SELECT u.id_user, u.u_age, u.u_deleted_date, u.u_name, u.u_phone
    FROM "user" AS u
    WHERE u.u_age >= @__age_0
    ```

    As you can see, all three requests point to line 30, even though each call actually occurs on a different line. Because of this, confusion may arise, and it is not always possible to quickly determine which request corresponds to which execution point.

    Using `.TagWithCallSite()` when creating an `IQueryable` is problematic because it captures where the query is created, not where it is executed. As a result, it becomes very difficult to determine exactly where a specific SQL query was executed in the logs.
