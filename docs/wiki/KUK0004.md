## 💡 Title
Kuk0004AvoidPrimaryConstructorAnalyzer

## 🆔 Rule ID
KUK0004

## 📝 Description
Avoid [primary constructor](https://learn.microsoft.com/en-us/dotnet/csharp/programming-guide/classes-and-structs/instance-constructors#primary-constructors)

## ❔ Why this is a problem
I think primary constructors are a bad feature because:

* Parameters aren't mutable unless you use various [hacks that fix this](https://stackoverflow.com/a/77831064/22817151), but in reality, these entries take up almost as much code and look worse, so it's pointless.

* They look like variables or method parameters, which can be confusing when used.

In short, the cons outweigh the pros, IMHO.

## ⚙️ Configuration

You can configure the severity of the rule using .editorconfig.
```.editorconfig
[*.cs]
dotnet_diagnostic.KUK0004.severity = warning # Avoid primary constructor | https://github.com/kurnakovv/kuker/wiki/KUK0004
``` 

And also disable [IDE0290](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/style-rules/ide0290) rule in .editorconfig file
```.editorconfig
# Use primary constructor (IDE0290)
csharp_style_prefer_primary_constructors = false
dotnet_diagnostic.IDE0290.severity = none
```

## 💻 Examples
```cs
// ❌ KUK0004: Avoid primary constructor for 'BadUserModel'.
public class BadUserModel(string name, int age)
{
    public string SayHello()
    {
        return $"Hello, I'm {name} and {age} y.o.";
    }
}

// ✅ OK
public class GoodUserModel
{
    private readonly string _name;
    private readonly int _age;

    public GoodUserModel(
        string name,
        int age
    )
    {
        _name = name;
        _age = age;
    }

    public string SayHello()
    {
        return $"Hello, I'm {_name} and {_age} y.o.";
    }
}
```

## 🗒️ Notes
* Records | This rule does not report a violation when primary constructors are used in records, for example: 
    ```cs
    public record Person(string FirstName, string LastName);
    ```
    In most cases, this usage is justified. Records are commonly used as DTOs, where the concise syntax of primary constructors provides meaningful benefits in readability and brevity.

* [IDE0290](https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/style-rules/ide0290) | Why not simply set `csharp_style_prefer_primary_constructors=false`? Because that option still allows both styles (primary constructors and traditional constructors).
This rule is stricter - it is intended to prohibit the use of primary constructors entirely.

* Classes and Structs | This rule applies to both classes and structs.
